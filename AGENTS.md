# AGENTS.md

Guidance for Codex and other AI coding agents working in this repository.

## Project Snapshot

`ncrc-app` is a real React 19 + TypeScript workflow app, not a starter template. It uses Vite, TanStack Router file routes, TanStack Query, Tailwind CSS 4, Radix UI primitives, Sonner toasts, and AWS Cognito OAuth with a local-dev login path.

Main authenticated workflows:

- Application Dashboard: `/ou-workflow/ncrc-dashboard`
- Tasks & Notifications: `/ou-workflow/tasks-dashboard`
- Application Intake Dashboard: `/ou-workflow/prelim-dashboard`
- Profile settings: `/profile`

Read `README.md` for architecture context before broad changes.

## Commands

- Install: `npm install`
- Dev server: `npm run dev` (Vite on port 3000)
- Typecheck: `npm run typecheck` or `npx tsc --noEmit`
- Test: `npm run test`
- Lint: `npm run lint`
- Build: `npm run build`

After TypeScript/route/API changes, run `npx tsc --noEmit` at minimum.

## Architecture Rules

- Keep route files thin. Route files should own route declaration, `validateSearch`, loaders, redirects, error boundaries, and mounting feature screens.
- Put endpoint wrappers in `src/features/<feature>/api`.
- Put query/mutation hooks in `src/features/<feature>/hooks`.
- Put query keys in `src/features/<feature>/model/queryKeys.ts`.
- Use `src/shared/api` helpers for transport, errors, query options, and query params.
- Avoid new imports from `@/api`; `src/api.ts` is a compatibility layer for older code.
- Prefer feature-owned UI under `src/features/*/components` and route-facing screens under `src/features/*/screens`.
- Treat `src/components/ou-workflow/*` as legacy/workflow-owned UI that is still mid-migration; avoid moving broad areas unless asked.

## Routing Notes

- File routes live under `src/routes`.
- Authenticated layout is `src/routes/_authed.tsx`; public auth layout is `src/routes/_public.tsx`.
- Large dashboards are lazy-loaded through `index.lazy.tsx` route files.
- Normalize search params in route `validateSearch`; do not rely on raw search values in feature code.
- `src/routeTree.gen.ts` is generated by TanStack Router. Do not hand-edit it unless there is no other option.

Important search contracts:

- NCRC Dashboard search:
  - `q: string`
  - `status: string`
  - `priority: string`
  - `page: number`
  - `myOnly: boolean`
  - `applicationId?: number`
- Task Dashboard search:
  - `qs: string`
  - `days: 'pending' | 7 | 30`
  - `page: number`

When linking to dashboards, provide the full required search object or a reducer that preserves existing values.

## API And Query Conventions

- Use `fetchWithAuth` from `src/shared/api/httpClient.ts`.
- Use `buildPaginationParams(page, limit)` from `src/shared/api/queryParams.ts` for paginated endpoints. It emits `page[limit]` and `page[offset]`.
- Preserve backend `meta` when endpoints return it; use `meta.total_count`, `meta.limit`, and `meta.offset` for pagination.
- Use `keepPreviousData` for paged React Query lists where page changes should not blank the UI.
- For infinite queries, compute the next offset from `lastPage.meta.offset + lastPage.meta.limit`.
- Invalidate feature query keys rather than broad unrelated keys when feasible.

Known paginated endpoints:

- Applications: `/get_applications_v1?page[limit]=...&page[offset]=...`
- Tasks: `/get_application_tasks?page[limit]=...&page[offset]=...`
- Prelim applications follow the same pagination helper pattern.

Known response quirks:

- Create dashboard app: `/createApplication?ownsId={id}` returns `{ application_id, status }`.
- `get_application_tasks` returns `{ data, meta, status }`.
- Some task records use mixed casing such as `TaskCategory`, `TaskInstanceId`, `taskCategory`, and `taskInstanceId`; normalize in mappers or feature helpers.
- Some backend fields may be the string `"NULL"`; UI helpers often treat that as empty/group assignment.

## Preferences And Pagination

User display preferences live in `AppPreferencesContext` and profile storage:

- `stageLayout`
- `paginationMode: 'paged' | 'infinite'`

Application Dashboard and Tasks & Notifications should respect `paginationMode`.

- Paged mode should call the backend with `page[limit]` and `page[offset]`.
- Infinite mode should fetch additional backend pages, not merely reveal a previously fetched full list.
- Reset dashboard `page` to `0` when search/filter values change.

## Domain Notes

- `applicationId` is the workflow application id.
- `companyId` is the account/company number in invoice contexts.
- Inspection invoice email and preview account number should use `companyId`, not `applicationId`.
- Task action handling is centralized in `src/features/tasks/hooks/useTaskActions.ts` and task dashboard state in `src/features/tasks/hooks/useTaskDashboardState.ts`.
- Application dashboard state is in `src/features/applications/hooks/useNcrcDashboardState.ts`.

## UI Guidelines

- Use existing Tailwind patterns and local components before adding new abstractions.
- Use `lucide-react` icons when an icon is needed.
- Keep operational dashboards dense, predictable, and work-focused.
- Avoid landing-page patterns for internal tools.
- Keep cards for repeated items, modals, and framed tools; do not nest UI cards inside cards.
- Preserve accessibility basics: buttons need `type="button"` when not submitting, dialogs need labels, disabled states should be clear.

## Testing And Verification

- Typecheck after most changes: `npx tsc --noEmit`.
- Run focused Vitest files when changing tested modules.
- Existing test coverage is light; add narrow tests when changing shared hooks, query behavior, mappers, or high-risk business logic.
- If a change affects frontend behavior, verify the route in the browser when practical.
